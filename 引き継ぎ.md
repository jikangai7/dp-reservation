# 引き継ぎ書：エアトリDP 予約入力画面への自動到達

別端末・別アカウントで作業を再開するための引き継ぎ資料です。
このフォルダ（`dp-reservation`）一式を新端末へ移し、下記「別端末セットアップ」を実施すれば続行できます。

> **フォルダの場所に注意**：当初 `C:\Claude\リベシティ\dp-reservation\` に作成したが、
> 途中でユーザーにより `C:\Claude\dp-reservation\`（リベシティ静的サイトとは別の独立フォルダ）
> に移動された。以降のパス例はこの新しい場所を前提にする。プレビュー起動用の
> `.claude\launch.json` はリベシティ側（`C:\Claude\リベシティ\.claude\launch.json`）の
> `dp-avail-runner` 設定が絶対パスで `C:\Claude\dp-reservation\server.js` を指している。

> 重要：このツールは**匿名SSO（ゲストセッション）**で動作し、ログイン不要です。
> したがって**アカウントが別でも影響ありません**。クッキー無しの新規ブラウザで毎回成立します。

---

## 1. これは何か / ゴール

エアトリ（skygate.co.jp）の**海外航空券＋ホテル（ダイナミックパッケージ, DP）**予約フローで、
**出発地・目的地・出発日・現地出発日（大人1名固定）の4項目**だけから、
予約者情報入力画面 `/dp/reservation/input` まで**自動で到達**させる。

- ゴール＝情報入力画面の**表示到達まで**。予約確定・決済はしない。
- 実現方式＝「URL手組み」ではなく「ブラウザでフローを順に駆動」する（理由は §3）。

---

## 2. これまでにやったこと（経緯）

1. `https://www.skygate.co.jp/dp/` を解析（Playwright MCPで操作）。DP検索フォームの構造を把握。
2. 検索を実行し、`/dp/searching` → `/dp/list` の遷移とURLパラメータを解明。
3. `/dp/list` の「この商品のフライトを見る」やフライト変更でURL変化（`selectedItemKey`）を確認。
4. 「予約確認へ進む」押下後の `/dp/reservation/input` のURL構造（`packageId` / `__tx__`）を分析。
5. AIR ONLY版（`/air/reservation/input`）と比較し、設計思想の違いを整理。
6. ネットワークを記録し、押下時に走る **`POST /dp/avail`** を特定。リクエスト/レスポンス本文まで取得。
7. 「4項目から到達できるか？」に対し、駆動方式で到達可能と結論。
8. 成果物（スクリプト＋手順書）を作成し、検証実行までPASSを確認。

## 3. 判明した技術ポイント（最重要・失うと再解析が必要）

### 到達フロー
```
4項目
  → GET /dp/searching?departure=&arrival=&destination=&fromDate=&toDate=&rooms=1&business=0&searchKind=0&AgentCode=HATOP
      （匿名SSO: sso_auth_check → account.airtrip.jp/auth → sso_callback を自動通過）
  → /dp/list 着地（おすすめ商品が自動選択＝selectedItemKey がURLに付く。手動選択不要）
  → 「予約確認へ進む」<a href="javascript:void(0);"> をクリック
      → XHR: POST /dp/avail（在庫確認+価格確定+packageId発行、約3秒）
      → 匿名SSO遷移ハンドオフで __tx__ 発行
  → GET /dp/reservation/input?__tx__=...&packageId=...   ← ゴール
```

### 中核エンドポイント：`POST /dp/avail`
- ヘッダ：`Content-Type: application/json`、`X-Requested-With: XMLHttpRequest`
- リクエストボディ（**価格は含めない**＝改ざん不可設計）：
  ```json
  {"controlKeys":{"AgentCode":"HATOP"},"condition":{"selectedItemKey":"..._dpci"},
   "localCondition":{"differentDate":0},"additionalInfo":"","memberOnlyDealsFlag":false}
  ```
- レスポンス：`availStatus`（air/hotel在庫）＋`amountInfo`（確定価格）＋`hotelInfo`/`airInfo`＋**`packageId`**

### 2つのトークン（なぜ静的URLを作れないか）
| トークン | 役割 | 発行元 | 性質 |
|---|---|---|---|
| `packageId` | 商品（便＋ホテル＋確定価格）の金庫番号 | `POST /dp/avail` のレスポンス | 使い捨て |
| `__tx__` | 予約手続きの通行証 | avail後の匿名SSO遷移ハンドオフ | 使い捨て・ワンタイム |
→ 4項目から計算不能。実行のたびに新規発行される（検証で毎回異なる値を確認）。

### 設計思想
- **探す段階＝ステートレス**（条件がURLに全部載る。`/dp/searching` は手組み可能）
- **買う段階＝ステートフル**（商品はサーバー保存、URLは参照票のみ。金額はURLに載せない）
- 参考：AIR ONLY版 `/air/reservation/input` は24パラメータで検索条件を保持し、人数は `adultNum`。
  DP版とは別設計（DP検索URLの人数パラメータ名は未確認 → 本ツールは大人1名限定）。

### /dp/list のキャッシュキー（追加調査で判明）
`/dp/list` のURLは3キーで構成され、性質が異なる：

| キー | 構造 | 入力から生成 |
|---|---|---|
| `hotelCacheKey` | `20_1_{fromYmd}_{toYmd}____{dest}__2__1__1___0_0_0_` | **可**（完全に入力由来。サーバー値と一致を実測） |
| `airCacheKey` | `0_{GUID}{ts}_1,25_{dep}_{arr}_{fromYmd}_{toYmd}_1__0_,,,0,0,0_{dest}_0_0` | **不可**（`{GUID}`はサーバーがフライト検索時に発番） |
| `selectedItemKey` | 例 `…_dpci` | **不可**（サーバー発番。省略すると一覧表示時に自動生成） |

実測で判明した挙動：
- `airCacheKey` は**検索セッションのクッキーに紐づかない**。別のクッキー無しブラウザでも
  キー値だけで `/dp/list` → 予約入力まで通る（＝抽出キーは持ち運べる）。
- `selectedItemKey` を付けずに `/dp/list` を開くと、サーバーがおすすめ商品を自動選択して生成する。
- `airCacheKey` は avail 時点のスナップショット。時間経過・在庫変動で失効しうる → `keys` で取り直す。
- **一覧のソートはURLパラメータだけで可能**：`airOrder=1`（料金が安い順）/`airOrder=2`（おすすめ順）
  ＋`form=sortAir&airPage=1`。この形式のURLは `airCacheKey` の代わりに
  `cacheKeyCode`（airCacheKeyのGUID+ts部分）＋明示的な検索条件で構成される。
  検索条件も airCacheKey に埋まっているため `parseAirCacheKey()` で全て抽出できる。
  実測で合計金額が昇順（¥65,688→¥84,828→…）になることを確認済み。
  ※ソートは表示順のみ。selectedItemKey（選択商品）は変わらない。

→ この知見により reserve.js に **`keys` / `open-list` / `proceed`** サブコマンド＋
  **`--sort cheap|recommend`** オプションを追加（§4）。keys は `listUrlCheapest`（安い順URL）も出力。
  「keys で本物のキーを抽出 → /dp/list で作業 → proceed で予約入力へ」という中間作業動線に対応。

### フライト選択とホテルソート（さらに追加調査で判明）
- **フライト選択はURL不可・クリック必須**：「この商品のフライトを見る」→
  「このフライトを選択する」(`a.choiceThisFlight`) で `form=reload_air` 再読込となり、
  URLの `selectedItemKey` が選択商品に更新される（裏でPOST /dp/choice）。
- **「ホテルを変更する」の実体はホテルタブへの切替**（フライト選択後に有効化）。
  タブ切替はURLで直行可能：`form=sortHotel&tab=hotel`。
- **ホテルのソートもURLだけで可能**：`hotelSort=1`オススメ/`2`価格の安い順(既定)/`3`高い順/`4`ランク順。
  必要な `checkin`/`checkout`/`destination` は airCacheKey から導出可能。
- airCacheKey 内のフラグはソート等で変化する（例 `_1__0_`→`_1__1_`）。
  `parseAirCacheKey()` の正規表現はこの区画を固定しないこと（固定して一度壊れた→修正済み）。

→ **`cheapest <dep> <dest> <from> <to>`** サブコマンドを追加：
  安い順→最安フライト選択→ホテル一覧（価格の安い順）を開いてURL出力まで自動実行。
  出力のホテルタブURLは `proceed --list-url` にそのまま渡して予約入力まで進める。

### 上位3ホテルのプラン選択・DOM構造・avail検証UI（さらに追加調査で判明）

- **ホテルカードの境界**は `div.boxSearchGeneric01`（`a.showHotelPlanList` の祖先）。
  **ホテル名は可視テキストになく**、同カード内 `a.openHotelMap[data-hotel_info]` の
  JSON属性（`JSON.parse(...).name.jp`）にのみ含まれる。安易にテキストマッチで拾おうとして
  一度失敗した（`h3,h4` 等のセレクタでは取れない）→ JSON属性パースに修正して解決。
- **ホテル代金・合計金額**は `li.origin`（`.col-l`見出し＋`.col-r`金額）と `li.total`
  （`.col-r`が合計、`.sub`が泊数表記）から取る。**textContent全体への正規表現は誤爆する**
  （「合計金額」と金額の間に「航空券1名+ホテル9泊1室」という数字入り文字列が挟まるため、
  `[^\d]*` 系の正規表現が途中の数字で止まってしまう）→ 各`<li>`要素を個別に見て解決。
- **プラン選択**：`a.showHotelPlanList`（`data-hotel_code`付き、Ajaxでカード内に展開）→
  展開後の最初の `a.choiceThisPlan`（img alt="このプランを選択する"）をクリックすると
  `form=reload_hotel` で再読込され `selectedItemKey` が更新される。
- **選択済みフライト情報の取得元**は `#selectedAir`（`h3.air-name`＝航空会社名、
  `dl.out-box`/`dl.home-box` の `dd p.go`/`p.in` に時刻＋`.data`に地名）。
  空港はIATA3文字コードでなく「大阪(関西国際空港)」のような日本語表記でのみ表示される。
- **カード表示金額とavail確定金額は一致しないことがある**：一覧カードの金額は
  「そのホテルの代表（多くは最安）プラン」、avail検証は「最初に表示されたプランを選んだ」
  結果のため、ホテルによっては代表プランと先頭表示プランが別物になりうる（バグではなく仕様）。
- **`POST /dp/avail` はCookie無しのcurlでも200 OKで通り、`packageId`まで取得できる**ことを
  実測で確認済み（`--data-binary @<jsonファイル>` 形式を推奨。ボディをコマンド引数に直接
  埋め込むとシェルのエスケープ差異で壊れやすい）。ただし続く `/dp/reservation/input` への
  到達には account.airtrip.jp 側のSPAシェル（JS実行必須）を経由するSSO遷移が必要なため、
  **curlだけでは到達確認できない**。OK/NG判定は必ずreserve.js（ブラウザ経由）で行う。
- **`packageId`/`__tx__` は発行直後であれば全く別の新規ブラウザセッションでも再利用できる**
  ことをPlaywrightで実証済み（同一URLに複数回navigateして毎回到達成功）。つまりWeb UIの
  「input画面を開く」ボタンは、一度到達したカードなら後から何度でも開き直せる
  （長時間経過で失効した場合のみ「avail実行」を再度押して新トークンを取り直す）。
- **`selectedItemKey`に無効値を渡してもサーバーが自動フォールバックしOKになることがある**
  一方、`airCacheKey`/`hotelCacheKey`自体が構造的に無効だと `/dp/list` が描画できず
  60秒タイムアウトでNGになる。avail-checkの「NG」は主にこの種の構造的失敗を検出するもの。

→ これらの知見により reserve.js に **`hotel-plans`**（ホテル「価格の安い順」「ホテルランクの
  高い順」各3件＝計6件のプラン選択＋情報収集＋1件1行の手動実行コマンド生成）と
  **`avail-check`**（1件を実行しOK/NG判定）を追加。さらに `server.js`＋`public/`
  （3層カード構成のローカルWeb UI、グループ見出し付き）を新規作成し、
  詳細な動作仕様は `仕様書.md` にまとめた。日付入力（出発日・現地出発日）はUIでは
  カレンダーピッカー（`type="date"`）で選択し、送信直前に `YYYY/MM/DD` へ変換している。

**サイト切替（SG / Travelko）機能を追加**（2026-07-04）：全コマンド共通の `--site sg|travelko`
（既定 `sg`）で `AgentCode`（`sg`=HATOP／`travelko`=HACTB）・`meta`（0／1）・`metaLandingInit`
（未付与／1）を切り替える。`meta`/`metaLandingInit` は `/dp/searching` と `/dp/list` 系URL
（`buildListUrl`/`buildSortedListUrl`/`buildHotelListUrl`）すべてに付与し、`/dp/avail` の
リクエストボディ（`controlKeys.AgentCode`）はAgentCodeのみ差し替え。`hotel-plans`の各エントリに
`site` を保存し、生成される `avail-check` コマンドにも `--site` を含めることで、収集時と同じ
サイトで再実行されるようにしている。Web UIには検索フォームに SG/Travelko ラジオボタン（既定SG）を
追加済み。**この機能はユーザー指示により実装直後の検証を保留しており、後日まとめて検証予定**
（検証手順.md の STEP 11 参照）。Travelkoパラメータ付きで実際に `/dp/reservation/input` まで
到達できるかは未確認。

    CLIで小規模実行→多め実行の順に2回連続実行し、前回分が正しく消えて今回分だけが
    残ることを確認済み。

## 4. 成果物一覧（このフォルダ）

reserve.js はサブコマンド方式（`run`＝既定 / `keys` / `open-list` / `proceed` / `cheapest` /
`hotel-plans` / `avail-check`）。詳細は README。

| ファイル | 内容 |

## 5. 別端末セットアップ（新アカウント・新PC）

### 5-1. ファイル転送
`dp-reservation` フォルダを新端末にコピー。**`node_modules` は不要**（サイズ大・OS依存のため）。
`transfer.zip`（同梱、node_modules除外済み）を展開してもよい。

置き場所は任意（例 `C:\work\dp-reservation`）。スクリプトは `__dirname` 基準なのでパスに依存しない。
※README/検証手順内の絶対パス例は自分の配置に読み替えること。

### 5-2. Node.js の用意
```powershell
node -v   # 出ればOK（v18+。動作確認は v26）
```
- 出ない場合：Node.js を https://nodejs.org からインストール（LTS可）。
- インストール済みだが PATH に無い場合（前端末がこれだった）：
  ```powershell
  $env:Path = "C:\Program Files\nodejs;" + $env:Path
  ```
  （新端末のインストール先に合わせて調整）

### 5-3. 依存導入（新端末で一度だけ）
```powershell
cd <dp-reservation の場所>
npm install
npx playwright install chromium
```

### 5-4. 動作確認
```powershell
node reserve.js OSA MNL 2026/10/01 2026/10/10
```
`REACHED: .../dp/reservation/input?...` が出て終了コード0ならセットアップ完了。
詳細な合否判定は `検証手順.md` を参照。

## 8. 注意点

- サイト仕様変更（セレクタ・パラメータ・SSO挙動）で動かなくなる可能性。まず `--headed` で目視し、
  失敗時に出る `error-<時刻>.png` と `FAILED at URL:` から原因箇所を特定する。
- `packageId`/`__tx__` は avail 時点のスナップショット。時間経過で失効しうる。
- 実行は相手サーバーへの実アクセス。過度な連打・大量実行は避ける（`/dp/avail` は約3秒の重い処理）。
- 本ツールのゴールは情報入力画面の表示到達まで。**予約確定・決済は絶対に自動化しない**方針。
