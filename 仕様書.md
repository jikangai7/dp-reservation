# 仕様書：avail 検証ランナー（エアトリDP予約フロー自動化ツール）

## 1. 目的

エアトリ（skygate.co.jp）の海外航空券＋ホテル（DP）予約フローにおいて、
**最安フライト選択 → ホテル「価格の安い順」「ホテルランクの高い順」それぞれ上位3件の先頭プラン選択 → `/dp/avail` 到達検証**
を自動化し、結果をローカルWeb UIで確認できるようにする。

ゴールは `/dp/reservation/input`（予約者情報入力画面）への**表示到達まで**。
予約確定・決済は一切行わない。

---

## 2. システム構成

```
┌────────────────────────┐
│ ブラウザ（利用者）         │
│  http://localhost:5178/ │
└───────────┬────────────┘
            │ HTTP (fetch)
┌───────────▼────────────┐
│ server.js（Node標準http） │
│  POST /api/collect       │
│  POST /api/avail-check   │
│  GET  /（静的配信 public/）│
└───────────┬────────────┘
            │ child_process.execFile
┌───────────▼────────────┐
│ reserve.js（CLI・Playwright）│
│  hotel-plans / avail-check │
│  （他: run/keys/open-list/  │
│    proceed/cheapest）      │
└───────────┬────────────┘
            │ ブラウザ自動操作
┌───────────▼────────────┐
│ www.skygate.co.jp（実サイト）│
└────────────────────────┘
```

- `server.js` は独自ロジックを持たず、**必ず `reserve.js` を子プロセスとして呼ぶ**
  （在庫確認・到達判定のロジックを二重化しないため）。
- `reserve.js` は毎回**新規Playwrightブラウザコンテキスト**（クッキー無し）を起動する。

## 3. データフロー：フライト巡回×ホテル2グループの収集（`hotel-plans` / `POST /api/collect`）

```
検索（4項目）
  → /dp/list 着地（おすすめ商品自動選択）
  → 料金が安い順に並び替え（airOrder=1）
  → 1ページ目のフライト一覧（最大25件）からキャリア名（h2 .airline-name）と
    商品番号（li.item-number）を収集
  → キャリアルールで処理対象フライトを決定：
      ・1位（最安）は常に対象
      ・直前に処理したフライトと同キャリアなら読み飛ばす
      ・同キャリアは合計2つまで（3つ目以降は読み飛ばす）
      ・1ページ目の最後まで確認（--max-flights N で対象数を制限可能）
      ・--carrier XX（キャリア2レター）指定時は検索自体が carriers=XX で
        そのキャリアに絞り込まれるため、上記ルールのグルーピングキーを
        「キャリア名」から「商品ID末尾1桁」に切り替える（同末尾1桁は合計2つまで）
  → 対象フライトごとに：
      安い順一覧へ戻る → 該当順位のフライトを選択
        「この商品のフライトを見る」→「このフライトを選択する」
        → selectedItemKey 更新（form=reload_air）
      → 選択済みフライト情報を収集（航空会社・便名・往復時刻・航空券代。
        便名は「フライト詳細を見る」展開後の #flightDetailOfSelectedItem .flight-number から取得。
        経由便は4便以上になることもある）
      → グループごとに（①価格の安い順 hotelSort=2 → ②ホテルランクの高い順 hotelSort=4。
        取得件数はグループ別に --top-cheap / --top-rank（0〜3、既定各1）で指定。
        0のグループはスキップ。両方0は指定不可）：
          ホテル一覧を該当ソートで開く＝そのグループの基準URL確立
          n=1..top について：
            基準URLへ戻る
            → n番目のホテルの「この商品のプランを見る」(a.showHotelPlanList) をクリック
            → 展開された最初のプランの「このプランを選択する」(a.choiceThisPlan) をクリック
            → selectedItemKey 更新（form=reload_hotel）
            → そのホテルのカード情報（名称・ホテル代・合計・泊数）とキー3種を記録
  → JSON出力（plans.json、entries＝対象フライト数×(top-cheap+top-rank)件）＋
    手動実行コマンド2種を1件1行で生成
```

### 収集される情報（1エントリ）

| フィールド | 内容 |
|---|---|
| `rank` | グループ内の順位（1〜3。グループごとにリセットされる） |
| `sortLabel` | そのエントリが属するグループ：「価格の安い順」または「ホテルランクの高い順」 |
| `flightRank` | そのエントリが属するフライトの安い順一覧での順位（1〜25） |
| `flight` | そのフライトの情報：airline / flightNumbers（経由便は4便以上のことも） / airFare / outbound / inbound / rank / **productId**（商品番号。フライト一覧カードの`li.item-number`から取得。無い場合はnull） |
| `hotel` | そのホテルの一覧カード表示：hotelCode / name / hotelFare / total / stay / **cancelInfo**（プラン展開時の`li.cancel`テキスト。キャンセル無料プランのときのみ値が入り、無料でなければnull） |
| `keys` | `selectedItemKey` / `airCacheKey` / `hotelCacheKey` |
| `listUrl` | プラン選択後の `/dp/list` URL |
| `availBody` | `/dp/avail` へ送信するリクエストボディ（JSON） |
| `commands` | `reserve`（reserve.jsコマンド）／`curl`（curlコマンド）の1行文字列。ファイル名（`avail-body-N.json`）はフライト・グループを跨いだ通し番号 |

`plans.json` 直下の `baseHotelUrls` は `{ cheap: <価格の安い順の基準URL>, rank: <ホテルランクの高い順の基準URL> }`。

**重要な注意**：`hotel.total` は「そのホテルの代表プラン（一覧表示上の金額）」であり、
`avail-check` で実際に検証される「最初に表示されたプランを選んだ結果」の金額と
**異なる場合がある**。実際の確定金額は avail 実行後のレスポンス（`amountInfo`）が正。

## 4. 「avail実行」ボタン押下時の動作仕様

Web UIの各カードにある「avail実行」ボタンを押すと、以下が順に実行される。

### 4.1 手順

1. **UI（クライアント）**：ボタン押下で該当カードのバッジを「実行中」（アンバー）に変更し、
   ボタンを無効化。`POST /api/avail-check` に `{ item, air, hotel, site }`（そのカードの
   `selectedItemKey` / `airCacheKey` / `hotelCacheKey`と、収集時に選んだサイト）を送信する。
2. **server.js**：`child_process.execFile` で
   `node reserve.js avail-check --item <item> --air <air> --hotel <hotel> --site <site>` を起動する
   （タイムアウト90秒）。
3. **reserve.js（avail-check）**：
   a. 新規Playwrightブラウザ（クッキー無し）を起動。
   b. 3キーから `/dp/list` のURLを組み立てて開く。
   c. 「予約確認へ進む」ボタンをクリック。
   d. これにより **`POST /dp/avail`** が実行される（在庫確認・価格確定・`packageId` 発行）。
   e. 続く匿名SSO遷移で **`__tx__`** が発行され、`/dp/reservation/input` へ遷移する。
   f. 遷移完了（タイムアウト60秒）を待ち、到達すれば成功、しなければ例外。
4. **判定**：
   - 到達できた → `RESULT: OK`。最終URL・`packageId`・`__tx__`・所要時間をメッセージ化。
   - 到達できない（タイムアウト等）→ `RESULT: NG`。理由とタイムアウト時間をメッセージ化。
5. **server.js**：`reserve.js` の標準出力を正規表現でパースし、
   `{ ok, finalUrl, packageId, tx, message }` をJSONでUIに返す。
6. **UI反映**：
   - バッジを OK（緑）／NG（赤）に更新し、メッセージ欄に結果を表示。
   - OKの場合のみ「input画面を開く」ボタンを表示（`href` に到達済みURLを設定、新しいタブで開く）。
   - カード下部のキー表（`POST /dp/avail` 行・`/dp/reservation/input` 行）に
     `packageId` / `__tx__` の実際の値を反映する（それまでは「未取得」表示）。

### 4.2 注意事項

- **実サーバー（skygate.co.jp）への実アクセス**を伴う。テスト用のモック環境ではない。
- **予約確定・決済は一切行わない**。到達＝情報入力画面の表示確認のみ。
- 1回の実行に **5〜15秒**（`/dp/avail` 自体の処理に約3秒、前後のSSO遷移・描画待ちを含む）。
- **実行のたびに新しい `packageId`／`__tx__` が発行される**（使い捨てトークン）。
  同じカードを再実行しても別の値になる。
- 各カードの状態（バッジ・finalUrl・packageId・tx）は**カードごとに独立して保持**され、
  他カードの実行結果で上書きされることはない。
- 「NG」は主に**通信・タイムアウト・構造的に不正なURL**を検出するものであり、
  `selectedItemKey` の指す商品単体が売り切れているかどうかを厳密に区別するものではない
  （サーバー側が自動的に代替商品へフォールバックしてOKになることがあるため）。

## 5. キー・トークンの整理（画面／APIごとの必要情報）

| 用途 | 必要なキー | 発行元 | 性質 |
|---|---|---|---|
| `/dp/list`（一覧画面） | `airCacheKey` / `hotelCacheKey` / `selectedItemKey` | 検索実行時にサーバーが発行 | 検索セッションのキャッシュ参照。クッキー非依存で持ち運び可能 |
| `POST /dp/avail`（在庫確認API） | 送信：`selectedItemKey`　応答：`packageId` | 送信は既知の値／応答はサーバー発行 | 価格は送信ボディに含めない（改ざん不可設計） |
| `/dp/reservation/input`（予約入力画面） | `packageId` / `__tx__` | avail応答／続くSSO遷移でそれぞれ発行 | 使い捨てだが、発行直後は別ブラウザ・別セッションでも再利用可能 |

### サイト別パラメータ（SG / Travelko）

`/dp/searching`・`/dp/list`（一覧画面）・`POST /dp/avail`（`controlKeys.AgentCode`）の
すべてで、選択したサイトに応じて以下のパラメータが変わる：

| サイト | AgentCode | meta | metaLandingInit |
|---|---|---|---|
| SG（既定） | HATOP | 0 | （付与しない） |
| Travelko | HACTB | 1 | 1 |

`hotel-plans` で収集した各エントリには `site`（`sg`/`travelko`）が保存され、
そのエントリの `avail-check` は自動的に収集時と同じサイトのパラメータで実行される
（サイトを跨いで混在させると `AgentCode` 不一致になるため）。

## 6. API仕様

### `POST /api/collect`

リクエスト：
```json
{ "dep": "OSA", "dest": "MNL", "from": "2026/10/01", "to": "2026/10/10", "site": "sg", "topCheap": 1, "topRank": 1, "carrier": "NH" }
```
- `dep`/`dest`：英字3文字の都市コード（`^[A-Z]{3}$`、小文字は自動で大文字化）
- `from`/`to`：`YYYY/MM/DD`
- `site`：`sg`（既定）または `travelko`
- `topCheap`/`topRank`：ホテル取得数（0〜3の整数、省略時は各1）。両方0は400エラー。
  タイムアウトは30分（フライト巡回のため）
- `carrier`：キャリア2レター（`^[A-Za-z0-9]{2}$`、省略可・空文字可）。指定時は検索自体を
  そのキャリアに絞り込み（`carriers=XX` を `/dp/searching` に付与）、フライト巡回のグルーピング
  キーが商品ID末尾1桁に切り替わる。レスポンスの `condition.carrier` に正規化後の値（未指定時は `null`）が入る

レスポンス（200）：`hotel-plans` の `plans.json` と同じ構造（`entries` 配列を含み、各エントリに `site` を含む）。
エラー時（400/502）：`{ "error": "..." }`

### `POST /api/avail-check`

リクエスト：
```json
{ "item": "<selectedItemKey>", "air": "<airCacheKey>", "hotel": "<hotelCacheKey>", "site": "sg" }
```
- `site`：`sg`（既定）または `travelko`。収集したエントリの `site` と一致させること。

レスポンス（200）：
```json
{ "ok": true, "finalUrl": "https://www.skygate.co.jp/dp/reservation/input?...", "packageId": "packageId_...", "tx": "...", "message": "/dp/reservation/input に到達（10.2秒）" }
```
`ok: false` の場合 `finalUrl`/`packageId`/`tx` は `null` になりうる。

## 7. 画面仕様

### カードの状態遷移

```
未実行（グレー） --[avail実行]--> 実行中（アンバー） --成功--> OK（緑）
                                                  \--失敗--> NG（赤）
OK / NG --[avail実行（再実行）]--> 実行中（アンバー） --...
```

### 画面上部の共通要素

- **検索フォーム**：出発地／目的地／出発日・現地出発日（カレンダーピッカー）／
  サイト（SG・Travelkoラジオ）／ホテル取得数（安い順0〜3・ランク順0〜3のプルダウン、既定各1）／
  「URL取得」ボタン。ボタン下に**取得経過時間**（押下〜カード表示完了まで1秒ごと更新、
  完了時「取得時間 MM:SS」で確定・緑表示。失敗時は非表示に戻る）
- **「CXL無料をすべて実行」ボタン**：`hotel.cancelInfo` があるカード（＝ホテルキャンセル料無料）
  のみを対象に順次avail実行。実行済み（OK/NG）カードはスキップ
- **サーバー停止ボタン**（画面右上）：確認ダイアログ後 `POST /api/server/stop` を呼び、
  サーバープロセスを終了。停止後は再起動案内バナーを表示し全操作を無効化
  （再起動は start-server.bat。UIから起動できないのはUI自体をこのサーバーが配信しているため）
- **残り時間タイマー**：1件目の `/dp/list` URL行が表示された時点から15分のカウントダウン
  （MM:SS表示・画面上部中央に固定・スクロール追従・0で赤表示）

### カード一覧の構成

一覧の一番上に1件目の `/dp/list` URL（コピー・開くボタン付き、`showtimeinfo=1` 付与）を表示し、
以降は**フライトごと**に「フライトN位 キャリア名 商品ID 航空券代」の見出し、
その下に「価格の安い順」見出し＋カードtop-cheap枚、「ホテルランクの高い順」見出し＋
カードtop-rank枚を繰り返し表示する（0件のグループは見出しごと非表示）。

### カード構成（上→下）

1. **ステータス行**：結果バッジ／メッセージ／「avail実行」ボタン／（OK時のみ）「input画面を開く」
2. **到達URL行**（OK時のみ表示）：`/dp/reservation/input` の到達済みURLをコピーボタン付きで表示
3. **AIR・HTLブロック（2カラム）**：見出し（ソート順ラベル＋順位バッジ）＋詳細。
   AIR側は常に「価格の安い順」ラベル（フライト一覧は安い順ソートで巡回するため。
   表示内容はそのカードが属するフライトの情報）、
   HTL側はそのカードが属するグループのソート順ラベルを表示する。
   - AIR側の表示順：キャリア名／便名 → **商品ID**（`productId`、無ければ非表示） →
     往復の時刻 → 航空券代
   - HTL側の表示順：ホテル名 → **ホテルID**（`hotelCode`） → ホテル代／泊数 → 合計金額 →
     **キャンセル無料表示**（`cancelInfo`、無料プランのときのみ表示）
4. **キー・トークン（折りたたみ）**：画面/API別キー表＋reserve.js/curlコマンド（コピー可）

## 8. FAQ／運用

**Q. カード①を到達確認した後、別のカードを実行した。後でカード①をもう一度開きたい。**
A. カード①の「input画面を開く」をもう一度押せばよい。`packageId`/`__tx__` は
発行直後であれば別セッション・別タブでも再利用できることを確認済み。
開けなくなっていた場合（長時間経過による失効）は、そのカードの「avail実行」を
再度押して新しいトークンを発行し直す（`selectedItemKey` は保持済みなので、
検索・プラン選択のやり直しは不要）。

**Q. curlコマンドだけで予約入力画面まで到達確認できるか。**
A. できない。`/dp/avail` 自体はCookie無しでcurlから200 OKが返り `packageId` も取得できるが、
続く `/dp/reservation/input` への到達にはブラウザでのSSO遷移（JavaScript実行）が必須。
そのため到達のOK/NG判定は必ず `avail-check`（reserve.js経由）で行う。curlコマンドは
「avail応答の生データを直接確認したいとき」の補助として提供している。

**Q. ホテルカードに表示される金額と、avail実行後の実際の金額が違う。**
A. 想定内。カード表示は一覧の代表プラン（多くは最安プラン）の金額、avail実行が検証するのは
「最初に表示されたプランを選んだ」結果の金額のため、ホテルによっては差異が出る。

---

## 9. 非対象

- 予約確定・決済（到達＝`/dp/reservation/input` 表示まで）
- 大人2名以上・複数部屋、ホテル絞り込み条件
- 4件目以降のホテル（既定 `--top 3`。オプションで変更可能）
- 2番目以降のプラン選択（各ホテルとも最初に表示されたプランのみ）
